## ROI repair order

ROI score uses this simple meaning:
- Impact: how much it improves quality, learning, or debugging.
- Effort: expected implementation difficulty.
- Risk: chance of breaking the pipeline or over-blocking good drafts.
- Priority: higher ROI means do earlier.

### ROI 1 — Generate and inspect `run_quality_report.json` on the next run

- [ ] Run the pipeline once on main after PR #139.
- [ ] Download `pipeline-diagnostics-<run_id>`.
- [ ] Confirm `run_quality_report.json` exists.
- [ ] Confirm status is one of `pass`, `fail`, or `inconclusive`.
- [ ] Confirm it reports sidecar count, detection count, track count, missing trace, missing dropped reasons, runtime bugs, and alerts.
- Impact: very high.
- Effort: low.
- Risk: low.
- Why first: without this, every future repair is judged manually and slowly.

### ROI 2 — Add minimal draft/source-window metadata to diagnostics

- [ ] Store one machine-readable file listing every uploaded draft.
- [ ] Required fields: draft filename/id, source video, selected start/end, source window start/end, QA status, review-required reasons, title/description, output path/link if available.
- [ ] Add it to diagnostics artifact.
- [ ] Extend `run_quality_report.json` to read it.
- Impact: very high.
- Effort: low-medium.
- Risk: low.
- Why second: mixed-subject and duplicate detection need source-window-to-draft mapping.

### ROI 3 — Detect obvious duplicate source-window overlap

- [ ] Compute overlap between draft source windows.
- [ ] Add metric `source_window_overlap_duplicate_rate`.
- [ ] Add classification `BUG_DUPLICATE_MOMENT_LIKELY` when two normal drafts overlap strongly.
- [ ] Do not block yet unless confidence is high; first mark fail/review-required in report.
- Impact: high.
- Effort: low.
- Risk: low.
- Why third: duplicate windows are easier to detect than same-athlete identity and directly match one observed failure type.

### ROI 4 — Detect mixed-subject risk using track dominance

- [ ] For each draft source window, compute visible track ids from sidecar detections.
- [ ] Compute `primary_track_dominance_ratio`.
- [ ] Compute non-primary visible duration/count/area proxy.
- [ ] Add `mixed_subject_likely_windows` to report.
- [ ] Trigger `BUG_MIXED_SUBJECT_LIKELY` when more than one significant track appears and primary dominance is below threshold.
- Impact: very high.
- Effort: medium.
- Risk: medium.
- Why fourth: it targets the most visible product failure, but thresholds need care to avoid blocking intentional social moments.

### ROI 5 — Add track-fragmentation metric and tracker tuning loop

- [ ] Add `track_fragmentation_rate` to report.
- [ ] Add distributions: track duration, detections per track, short-track rate.
- [ ] Trigger `BUG_TRACKING_FRAGMENTATION_LIKELY` when short-track rate is high.
- [ ] Create follow-up tuning experiments for `vid_stride`, tracker config, confidence threshold, and model size.
- Impact: high.
- Effort: medium.
- Risk: medium.
- Why fifth: duplicate-athlete detection is unreliable until fragmentation is understood.

### ROI 6 — Make QA blocking policy explicit

- [ ] Convert critical QA defects into hard-block/review-required policy.
- [ ] Critical defects: `IDENTITY_MISMATCH`, mixed-subject signal, duplicate moment, missing source evidence, cut-too-early.
- [ ] Add metric `qa_gate_bypass_rate`.
- [ ] Trigger `BUG_QA_GATE_BYPASSED` when critical defect exists but draft is normal/approvable.
- Impact: high.
- Effort: medium.
- Risk: medium-high.
- Why sixth: it prevents bad drafts from looking normal, but policy must avoid blocking every useful draft.
